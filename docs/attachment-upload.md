# IETM Attachment Upload

## Overview

The IETM Playwright Client now supports uploading test execution output as text attachments to IETM execution results. This feature is based on the Java reference implementation and uses multipart/form-data for reliable file uploads.

## Implementation Details

### Java Reference Analysis

The implementation is based on the Java adapter's `RqmClient.createAttachment()` method:

```java
// Java reference implementation
HttpEntity data = MultipartEntityBuilder.create()
    .setMode(HttpMultipartMode.BROWSER_COMPATIBLE)
    .addBinaryBody("upfile", file, contentType, file.getName())
    .build();

String url = getRqmBasePath() + "/attachment/urn:com.ibm.rqm:attachment";
HttpUriRequest request = RequestBuilder.post(url).setEntity(data).build();

// Get attachment URL from Location header
String locationHeader = response.getFirstHeader("Location").getValue();
```

### Key Features

1. **Multipart/Form-Data Upload**: Uses the field name "upfile" as per IETM API requirements
2. **Location Header**: Retrieves the attachment URL from the HTTP Location header
3. **Text Attachments**: Uploads Playwright test execution output as formatted text files
4. **Automatic Integration**: Seamlessly integrated into the IETMReporter

## Architecture

### AttachmentHandler

The `AttachmentHandler` class provides the core attachment upload functionality:

```typescript
class AttachmentHandler {
  /**
   * Upload test execution output as text attachment
   */
  async uploadTestOutput(
    testOutput: string,
    testName: string,
    executionResultId: string
  ): Promise<string | null>

  /**
   * Upload a file to IETM
   */
  async uploadFile(
    executionResultId: string,
    filePath: string,
    description?: string
  ): Promise<UploadResult>
}
```

### IETMReporter Integration

The reporter automatically uploads test output for each test execution:

```typescript
private async uploadAttachments(
  executionResultId: string,
  result: CollectedTestResult
): Promise<void>

private buildTestOutput(result: CollectedTestResult): string
```

## Test Output Format

The uploaded text attachment includes:

1. **Header**: Test name, status, duration, start time
2. **Location**: File path and line number
3. **Error Details**: Error message and stack trace (if failed)
4. **Test Steps**: Individual step results with durations
5. **Standard Output**: Console output from the test
6. **Standard Error**: Error output from the test
7. **Artifacts**: List of screenshots, videos, and traces
8. **Footer**: Generation timestamp

## Usage

### Automatic Upload (via Reporter)

The reporter automatically uploads test output when configured:

```typescript
// playwright.config.ts
import { IETMReporter } from 'ietm-playwright-client';

export default {
  reporter: [
    ['list'],
    [IETMReporter, {
      configPath: 'config/ietm.config.json',
      enabled: true,
      uploadScreenshots: true,  // Include screenshot info in output
      uploadVideos: true,       // Include video info in output
      uploadTraces: true,       // Include trace info in output
    }]
  ],
};
```

### Manual Upload

You can also upload attachments manually:

```typescript
import { IETMClient } from 'ietm-playwright-client';
import { AttachmentHandler } from 'ietm-playwright-client/attachments';

// Initialize client
const client = new IETMClient(config);
await client.initialize();

// Create attachment handler
const attachmentHandler = new AttachmentHandler(client);

// Upload test output
const testOutput = `
Test: My Test
Status: passed
Duration: 1000ms
...
`;

const attachmentUrl = await attachmentHandler.uploadTestOutput(
  testOutput,
  'My_Test',
  'urn:com.ibm.rqm:executionresult:123'
);

console.log('Attachment URL:', attachmentUrl);
```

## API Endpoint

The attachment upload uses the following IETM endpoint:

```
POST {baseUrl}/qm/service/com.ibm.rqm.integration.service.IIntegrationService/resources/{projectArea}/attachment/urn:com.ibm.rqm:attachment
```

**Request:**
- Method: POST
- Content-Type: multipart/form-data
- Field name: "upfile"
- Accept: application/xml

**Response:**
- Status: 201 Created
- Location header: Contains the attachment URL

## Troubleshooting

### Common Issues

1. **HTTP 415 Unsupported Media Type**
   - Ensure you're using multipart/form-data
   - Verify the field name is "upfile"

2. **HTTP 400 Bad Request**
   - Check that the content type is set correctly
   - Verify the file content is valid

3. **No Location Header**
   - The upload may have failed
   - Check server logs for details

4. **Authentication Errors**
   - Verify credentials in config file
   - Ensure user has permission to upload attachments

### Debug Logging

Enable debug logging to see detailed upload information:

```typescript
const attachmentHandler = new AttachmentHandler(client);

// The handler logs debug information automatically

```

## Limitations

1. **Text Only**: Currently supports text attachments only (not binary files like images)
2. **Size Limit**: Default maximum file size is 10MB
3. **No Direct Linking**: Attachments are uploaded but not automatically linked to execution results in the XML
4. **Step Results**: IETM typically links attachments to step results, not execution results directly
